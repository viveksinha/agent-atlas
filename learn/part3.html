T. Subagents — Delegating Work to Specialized Agents

A subagent is a smaller agent that a main agent spins up to handle one piece of a larger task. The main agent stays in charge; the subagent does focused work and reports back.

Think of it as a manager delegating to specialists instead of doing everything alone.

One agent vs subagents

One agent does everythingMain agent + subagents
Reads 50 sources, writes report, validates dataMain agent assigns: "you search filings," "you pull metrics," "you write the summary"
Context fills up fastEach subagent gets only what it needs
Slower on complex tasksParallel work — multiple subagents can run at once

Simple analogy: One person cooking a five-course dinner vs a head chef coordinating saucier, pastry chef, and grill cook.

When subagents help

SituationWhy subagents work
Large market landscape scanOne subagent searches news, another SEC filings
Research + writingOne subagent gathers facts, another drafts the brief
Parallel checksOne subagent validates data quality, another reviews source coverage
Unfamiliar domainA specialized subagent gets a tighter prompt and focused tools

How it works in practice

flowchart TD

    Main --> A[Subagent A\nsearch filings & news]
    Main --> B[Subagent B\npull market metrics]
    A --> Synth[Main agent\nsynthesizes findings]
    B --> Synth
    Synth --> Propose[Proposes fix]
    Propose --> Approve{You approve?}
    Approve -->|Yes| Implement[Main or coding\nsubagent implements]

Example in Cursor: You ask for a broad competitive landscape audit. The main agent launches explore-type subagents to search filings, news, and internal dossiers in parallel, then combines their findings into one report — instead of researching each source sequentially in one thread.

Subagents vs skills vs workflows

SkillWorkflowSubagent
What it isWritten playbookSequence of stepsSeparate AI worker
Best forRepeatable procedure you definedMulti-step job you run on scheduleExploration, parallel work, heavy reasoning
RunsInside one agentOne agent, many stepsMultiple agents, coordinated
Example"How to write a competitive pulse brief""Weekly digest: Microsoft Teams → summary → Drive""Search all competitor filings for pricing changes"

Rule of thumb: Use a skill when you know the steps. Use a workflow when the steps are long but linear. Use subagents when the task is wide, exploratory, or benefits from parallel specialists.

U. Hooks & Automations — Running Agents Without Asking

So far, you've started agents by typing a prompt. Automations run agents on a trigger — an event or schedule — without you being in the chat.

A hook is a small script that fires when something happens (you save a file, open a PR, push code) and tells an agent to do a predefined job.

Trigger types

TriggerFires when…Example use
ManualYou type a prompt"Summarize this doc"
File saveYou save a specific fileAuto-format markdown, update a table of contents
Git eventPR opened, commit pushedRun a review checklist, update changelog
ScheduleCron / timer (daily, weekly)Monday 9am competitive pulse report
WebhookExternal system pings youNew research request → triage agent

Simple analogy: A prompt is calling a contractor when you need them. An automation is a standing order — "every Monday, do X" or "whenever this door opens, do Y."

Example: PR review hook

Example: PR review hook

sequenceDiagram
    participant Dev as Developer

The agent doesn't merge on its own — it prepares the review. Human-in-the-loop still applies (Part 2, Section M).

Building a safe automation

Before turning anything on unattended, check:

QuestionSafe if…
Can this break production?No — read-only or draft-only actions
Can it spam people?No — posts only with approval or to a private channel
Can it leak secrets?No — rules block .env, credentials, PII
Does it need fresh context every run?Yes — skill tells it where to pull data each time
Have you tested 3 real runs?Yes — Part 2, Section R

Practical rule: Automate preparation first (drafts, summaries, checks). Automate execution (send, publish, deploy) only after weeks of reliable drafts.

V. RAG — Giving Agents a Searchable Knowledge Base

RAG = Retrieval-Augmented Generation. It is a pattern where the agent searches a knowledge base first, then generates an answer grounded in what it found.

Without RAG, the agent only knows what's in its training data and your current context. With RAG, it can pull from your company's docs, wiki, past research requests, or document index.

How RAG works

flowchart LR
    Q[Your question] --> Search[Search knowledge base\nwiki, dossiers, data dictionary]
    Search --> Chunks[Retrieve top\nrelevant chunks]
    Chunks --> Context[Add chunks\nto context]
    Context --> Gen[LLM generates\n grounded answer]
    Gen --> A[Company-specific\nresponse]

    style Search fill:#cce5ff,stroke:#333
    style Gen fill:#e8f4fc,stroke:#333
    style A fill:#d4edda,stroke:#333

This is the classic Retrieve → Augment → Generate pipeline. Search first, then answer from what was found — not from memory alone.

Simple analogy: Open-book exam vs closed-book. RAG lets the agent open the book (your docs) before answering.

RAG vs putting files in context

@-mention / open filesRAG
You chooseYes — you pick exact filesAgent searches — finds relevant sections automatically
ScaleWorks for a few filesWorks for thousands of docs
Best for"Use this spec" / "Fix this file""What do we know about X across the org?"
RiskLow — you control what's includedMedium — wrong chunk retrieved → wrong answer

What makes a good knowledge base for RAG

Good sourcePoor source
Up-to-date wiki pagesDuplicate docs with conflicting info
Tagged support macrosRaw Microsoft Teams threads with no structure
Single source-of-truth policiesPDF scans with no text layer
README and architecture docsOutdated docs nobody maintains

RAG in everyday agent work

You may already use RAG without the label:

  • Document indexing in Cursor — agent searches your project files semantically
  • Confluence / Drive search via MCP — agent queries connected knowledge
  • "Search company knowledge" skills — playbook that says: search internal docs first, then answer

Example: "How do we source pricing data for a new competitor?" The agent searches your internal research playbook and data dictionary, then summarizes the steps — instead of guessing from generic internet knowledge.

When RAG fails

ProblemFix
Answer cites outdated policyMaintain docs; add "last reviewed" dates
Agent can't find the docImprove titles, headings, and keywords in source docs
Answer mixes two conflicting docsConsolidate sources; mark one as canonical
Retrieves irrelevant chunksNarrow the search (specific folder, competitor, or tag)

Rule of thumb: RAG is only as good as the documents you feed it. Garbage in, garbage out — same as any search engine.

W. Cost & Tokens — Keeping Agent Runs Affordable

Every agent run consumes tokens — the units LLMs use to measure text in and text out (see Part 1, Section G). More tokens = more cost and often more time.

Understanding cost helps you build agents that are effective and sustainable — especially for automations that run daily or across a team.

What uses tokens

Consumes tokensUsually free / local
Your prompt and the agent's repliesReading a file from disk (before content goes to the model)
File contents loaded into contextMCP auth handshake
Tool outputs returned to the agentRules and skills stored on disk (only loaded portions count)
Long conversation history

Rough math: ~1 token ≈ 0.75 words. A 10-page doc ≈ 7,500–10,000 tokens. Loading 20 files into one chat can burn through a large context window in a single run.

Cost drivers in agent workflows

DriverWhy it costs moreHow to reduce
Large contextWhole repo or long docs in every run@-mention only relevant files; use RAG search instead of dumping everything
Reasoning modelsSlower, pricier, more thoroughUse for hard tasks only; use fast models for formatting and simple edits
Many tool callsEach call's output goes back into contextCombine steps; write skills that avoid redundant searches
Long agent loopsAgent retries, explores, backtracksClear prompt with done criteria; Plan mode before Agent mode
Automations at scale50 runs/day × team of 20Cache results, batch weekly instead of hourly, read-only summaries

Model choice vs cost (practical guide)

Task tierModelCost mindset
Tier 1 — trivialFast / lightweightPennies; don't overthink
Tier 2 — daily workStandard balancedDefault for most agent tasks
Tier 3 — hard problemsReasoning / large contextUse deliberately; worth it when stuck
Tier 4 — automated at scaleOften Tier 1 or 2 with tight skillsDesign the workflow to minimize tokens per run

Example: A weekly report automation that loads 100 files every Monday is expensive. Better: a skill that searches Microsoft Teams and Jira for last week's date range and pulls only matching items — maybe 5–10 snippets instead of 100 files.

Signs your agent is too expensive

SignalLikely cause
Runs take minutes for simple tasksWrong model or too much context
Same search repeated every stepSkill doesn't cache or summarize early
Automation bill spikesTrigger fires too often or on every file save
Context overflow mid-taskAgent re-reads everything after each tool call

Rule of thumb: Before automating, estimate: How many tokens per run × how many runs per month? If you wouldn't run the same query manually 500 times, tighten the workflow first.

X. Team Rollout — Sharing Skills, Rules, and Agents

Once one person has a working agent setup, the next step is making it useful for the whole team — without everyone reinventing prompts or breaking guardrails.

Team rollout means shared standards (rules), shared playbooks (skills), and shared integrations (MCP) — so every agent behaves consistently.

What to share vs keep personal

Share at team levelKeep personal
Rules (security, approval, tone)Your individual @-mentions and chat history
Skills for recurring workflowsExperimental one-off prompts
MCP connections (with admin oversight)Private notes and drafts
AGENTS.md or project README for agent contextPersonal API keys

Layers of shared configuration

LayerWhat it containsExample
Org-wide rulesSecurity, compliance, brand voice by the way Facebook marketplace Display why do you think"Never include customer names in external drafts"
Project rulesTeam-specific conventions"Always cite sources in competitive briefs; use ISO dates in reports"
Team skillsStandard workflowscompetitive-pulse, stakeholder-requirements, weekly-metrics-digest
Personal skillsYour shortcuts"My weekly metrics recap format"

Simple analogy: Rules = company policy. Project rules = team norms. Skills = department SOPs. Personal skills = your own cheat sheets.

Rollout checklist

StepAction
1. PilotOne person runs the workflow 3+ times successfully
2. DocumentTurn the workflow into a skill with triggers, steps, and output format
3. HardenAdd rules for approval, secrets, and failure handling
4. ShareCommit skills and .md playbooks to a shared repo or skills library
5. Train30-minute walkthrough: when to use it, what it won't do, how to review output
6. MeasureTrack time saved, error rate, and override frequency for 2–4 weeks
7. IterateUpdate the skill based on real failures — not one person's preference

Common rollout mistakes

MistakeBetter approach
"Everyone use AI for everything"Start with 2–3 high-pain, high-repeat workflows
No review stepHuman-in-the-loop on all external actions
Skills live in one person's laptopVersion-controlled shared skills folder
Stale docs in RAGAssign doc owners; review quarterly
Different MCP access per personStandardize connectors; IT approves scopes

Example: A competitive intelligence and analytics team shares a weekly-intel-digest skill. Everyone's agent pulls from the same Microsoft Teams channel and data warehouse export, uses the same section headings, and follows the rule "draft only — never post to the leadership channel in Microsoft Teams without explicit send." New analysts get consistent output on day one.

Y. The Production Agent Checklist — From Experiment to Trusted System

This section ties Part 3 together. Use it before you call any agent "production ready" — whether it runs on a schedule, serves a team, or touches customer data.

Before you ship

AreaCheck
PurposeOne sentence: what job does this agent do, and for whom?
TriggerManual, hook, or schedule — documented
SkillStep-by-step playbook exists and is version-controlled
RulesApproval gates, secret handling, no-spam policies in place
ToolsOnly the MCP servers and permissions needed — nothing extra
ModelRight tier for the task; reasoning model only if justified
CostEstimated tokens per run × runs per month is acceptable
Evaluation3/3 real test runs passed (Part 2, Section R)
Failure modeYou know what happens when search fails, API is down, or context overflows
OwnerNamed person who updates the skill when the workflow changes

Maturity levels

LevelDescriptionExample
0 — ExperimentYou try prompts ad hoc"Let me ask Claude about this bug"
1 — RepeatableSkill + rules, manual trigger"Every Friday I run the competitive-pulse-brief skill"
2 — Assisted automationScheduled or hooked, human approves output"Monday draft appears in Drive; I edit and send"
3 — Trusted automationRuns end-to-end with monitoring"Nightly sync creates draft Jira tickets from new research requests"
4 — Team platformShared skills, RAG, governance"All analysts use the same intel digest skill on shared MCP"

Most teams should live at Level 1–2 for months before jumping to Level 3. Level 4 is worth it when the same workflows are duplicated across dozens of people.
