J. Tools — How Agents Take Action in the Real World

A tool is anything an agent can use beyond plain text — read a file, run a command, search the web, post to Microsoft Teams, create a Jira ticket.

Without tools, an agent is just a chatbot. With tools, it can do things in your systems.

Common tool types

Tool typeWhat it doesExample
File systemRead, write, search files on your computer"Find all .md dossiers that mention Competitor X"
Terminal / shellRun commands (git, npm, python scripts)"Run the test suite and tell me what failed"
Web searchLook up current information"What did our competitor announce this week?"
APIsTalk to external servicesCreate a ticket, send an email, fetch analytics
BrowserVisit and interact with web pages"Check if the login page loads correctly"

Simple analogy: The LLM is the brain. Tools are the hands. The brain decides what to do; the tools do it.

Brain + hands architecture

    Brain -->|calls| Hands[Tools — the hands]
    Hands --> FS[File system]
    Hands --> Shell[Terminal]
    Hands --> Web[Web search]
    Hands --> API[APIs]
    Hands --> Browser[Browser]
    FS & Shell & Web & API & Browser -->|results| Brain

How an agent uses tools (the loop)

sequenceDiagram
    participant You
    participant Agent as Agent (LLM)
    participant Tool

    You->>Agent: Give a task
    Agent->>Agent: Think: need workshop notes
    Agent->>Tool: read_file("notes.md")
    Tool-->>Agent: File contents → added to context
    Agent->>Agent: Think: check Jira for duplicates
    Agent->>Tool: search_jira("competitor pricing")
    Tool-->>Agent: Search results → added to context
    Agent->>You: Task complete

Each tool call adds its result to the context (see Part 1, Section G). The agent reads those results and decides the next step.

Note: Workshop notes, skills, and briefs are commonly stored as .md files because Markdown is plain text that both you and the agent can read reliably in VS Code or Cursor.

K. MCP — Connecting Agents to Your Apps

MCP = Model Context Protocol. It is a standard way for AI agents to connect to external apps and services — Microsoft Teams, Jira, Google Drive, GitHub, and many others.

Think of MCP as a universal adapter. Instead of every AI tool building custom integrations for every app, MCP gives them one shared plug format.

Without MCP vs With MCP

Without MCPWith MCP
Agent can only read local filesAgent can read Microsoft Teams messages, Jira tickets, Drive docs
You copy-paste data into the chatAgent pulls data directly from the source
Each integration is custom and fragileOne protocol, many connectors

What MCP servers give an agent

An MCP server is a connector for one service. When enabled, the agent gets new tools like:

  • search_teams — find messages in a channel
  • create_jira_issue — open a new ticket
  • read_gdrive_file — fetch a document from Google Drive
  • list_github_prs — see open pull requests

Example in Cursor: You say "Summarize what the team flagged about Competitor X in #competitive-intel this week." The agent uses the Microsoft Teams MCP server to search the channel, reads the messages, and writes a summary — without you pasting anything.

Analogy: MCP is like giving your intern login access to Microsoft Teams, Jira, and Drive — but only the specific actions you've approved.

MCP architecture (universal adapter)

flowchart LR
    Agent[AI Agent\nCursor, etc.] <-->|MCP protocol| MCP[MCP layer]
    MCP <--> Teams[Microsoft Teams]
    MCP <--> Jira[Jira]
    MCP <--> Drive[Google Drive]
    MCP <--> GH[GitHub]

    style MCP fill:#e8f4fc,stroke:#333
    style Agent fill:#d4edda,stroke:#333

One standard plug (MCP) on the agent side. Many connectors on the app side. Add a new service = add a new MCP server, not a whole new integration.

L. Models — Picking the Right Brain for the Job

Not all LLMs are the same. Models differ in speed, cost, reasoning ability, and context window size. Choosing the right one affects quality and how long tasks take.

The main trade-offs

Model typeStrengthsBest for
Fast / lightweightQuick responses, lower costSimple edits, short drafts, quick questions
Standard / balancedGood all-around qualityMost day-to-day agent work, coding, writing
Reasoning / thinkingDeep analysis, multi-step logicArchitecture decisions, complex bugs, planning
Large contextHolds more files and history in memoryLong reports, multi-source dossiers, large datasets

Examples in Cursor: GitHub Copilot (fast), Claude Sonnet (balanced), Claude Opus or o3 (deeper reasoning, slower).

When to switch models

SituationModel choice
"Fix this typo"Fast model — don't overthink it
"Refactor this module across 10 files"Standard or large-context model
"Why does our auth flow fail intermittently?"Reasoning model — needs careful analysis
"Explore an unfamiliar dataset archive"Large-context model — needs to hold many files and tables

Rule of thumb: Start with a balanced model. Switch to a reasoning model when the agent keeps getting stuck or makes logical errors. Switch to a fast model for repetitive, simple tasks.

Model selection flow

flowchart TD
    Start[New task] --> Q1{Simple edit\nor quick question?}
    Q1 -->|Yes| Fast[Fast / lightweight model]
    Q1 -->|No| Q2{Needs deep reasoning\nor architecture?}
    Q2 -->|Yes| Reason[Reasoning / thinking model]
    Q2 -->|No| Q3{Many files or\nlong documents?}
    Q3 -->|Yes| Large[Large-context model]
    Q3 -->|No| Balanced[Standard / balanced model]

M. Human-in-the-Loop — When to Approve, When to Auto-Run

Human-in-the-loop means you stay involved at key decision points instead of letting the agent run completely unsupervised.

Not every action needs your approval. The goal is to approve the risky steps and automate the safe ones.

Autonomy levels

LevelWhat happensExample
Full supervisionAgent proposes every step; you approve before it runs"Here's the SQL I'll run — OK to proceed?"
Checkpoint approvalAgent runs safe steps freely; pauses before irreversible actionsReads files freely, asks before pushing to git
AutonomousAgent completes the whole task; you review the result"Summarize this doc" — nothing to break
Scheduled / backgroundAgent runs on a timer with no you in the roomWeekly report every Monday at 9am

What usually needs approval

Safe to auto-runNeeds approval
Read files, search, analyzeDelete files, drop database tables
Draft text (email, brief, report)Send email, post to Microsoft Teams, publish
Run tests locallyPush code, merge PRs, deploy
Create draft ticketsCreate tickets assigned to others without review

This is exactly what Rules are for (Part 1, Section E): "Never post to Microsoft Teams without explicit permission" is a human-in-the-loop guardrail baked into the agent.

Practical rule: Auto-run read and draft steps. Require approval for anything that leaves your machine or affects other people.

Autonomy spectrum

flowchart LR
    L1["Full supervision\nevery step approved"] --> L2["Checkpoint approval\npause before risky actions"]
    L2 --> L3["Autonomous\nreview output after"]
    L3 --> L4["Scheduled / background\nruns without you"]

    style L1 fill:#f8d7da,stroke:#333
    style L2 fill:#fff3cd,stroke:#333
    style L3 fill:#cce5ff,stroke:#333
    style L4 fill:#d4edda,stroke:#333

Move right only as trust grows. Most production agents should live at Level 2 until evaluation passes (Section R).

N. Building Your First Skill

A skill is a markdown file that teaches the agent how to do one specific job, step by step. You write it once; the agent follows it every time that task comes up.

Why Markdown (.md) files?

Markdown is a plain-text document format. You write headings with #, bullets with -, and bold with text. The file extension is .md.

Agent tools like Cursor and editors like VS Code standardize on Markdown for skills, rules, and project docs — not Word, not PDF.

ReasonWhy it matters
LLMs understand plain textNo special export step; the agent reads the file directly
Structure maps to steps# Steps and numbered lists match how skills and checklists work
Native in VS Code / CursorSyntax highlighting, search, and a Markdown Preview side panel
Git-friendlyDiff two versions of a skill, review changes in a PR, roll back mistakes
Same format everywhereSkills (SKILL.md), notes (workshop-notes.md), project context (README.md, AGENTS.md)
flowchart LR
    You[You edit .md file\nin VS Code / Cursor] --> File[Plain text\non disk]
    File --> Agent[Agent reads file\ninto context]
    File --> Preview[Preview pane\nfor humans]

    style File fill:#fff3cd,stroke:#333
    style Agent fill:#e8f4fc,stroke:#333

Simple analogy: Markdown is a shared notebook format — you read it normally, the editor can prettify it, and the AI doesn't need a translator.

In VS Code / Cursor: Open any .md file → open Preview (Cmd+Shift+V on Mac, Ctrl+Shift+V on Windows) to see headings and lists rendered cleanly while the agent still sees the raw text.

Anatomy of a skill

SectionPurposeExample
TriggerWhen should the agent use this skill?"Use when the user asks for a competitive pulse brief"
StepsOrdered checklist the agent follows1. Pull competitor name → 2. Find key market move → 3. Write 3 summary angles
Output formatWhat the finished work looks like"One-page brief with executive summary and source links"

Example skill (simplified)

# Write a Competitive Pulse Brief

Use when the user asks for a brief on a competitor's recent market move.

## Steps
1. Ask for the competitor name and time window if not provided.
2. Pull key signals: pricing changes, partnerships, filings, or news.
3. Write 3 brief angles: threat-led, opportunity-led, and watch-list-led.
4. Recommend the clearest angle for executives and explain why.

## Rules
- Always cite source URLs or document names.
- Never state unverified claims as fact.

Where skills live

In Cursor, skills are stored as SKILL.md files in your skills folder. The agent reads the description and loads the full skill when your task matches.

You don't need to learn all of Markdown on day one — headings, numbered lists, and bullet points cover most skills. VS Code and Cursor autocomplete # headings and list syntax as you type.

Analogy: Writing a skill is like onboarding a new hire for one recurring job. You explain it once; they do it the same way every time.

O. When Things Go Wrong — Common Agent Failures

Agents are powerful but not perfect. Knowing the common failure modes helps you fix problems faster instead of blaming the AI.

The big five

FailureWhat it looks likeWhat to do
HallucinationAgent states facts, file paths, or API responses that don't existAsk it to show its source. Use "read the file and quote the exact line."
Context overflowAgent "forgets" earlier instructions or files mid-taskShorten the task. @-mention only the files needed. Start a fresh chat.
Wrong tool choiceAgent searches the web when it should read a local file (or vice versa)Be explicit: "Read config.yaml in this repo — do not search online."
Partial completionAgent does 3 of 5 steps and says "done"Give a checklist in your prompt: "Complete all 5 steps before responding."
Over-eager editsAgent refactors code you didn't ask it to touchUse Ask or Plan mode first. Add a Rule: "Only edit files directly related to the task."

Debugging checklist

flowchart TD
    Bad[Agent gave a bad result] --> C1{Right files\nin context?}
    C1 -->|No| Fix1["@-mention missing files"]
    C1 -->|Yes| C2{Prompt specific\nenough?}
    C2 -->|No| Fix2["Add role, task,\ncontext, format"]
    C2 -->|Yes| C3{Right tools\nused?}
    C3 -->|No| Fix3["Name tool or\ndata source explicitly"]
    C3 -->|Yes| C4{Context\nfull?}
    C4 -->|Yes| Fix4["Fresh chat,\nfewer files"]
    C4 -->|No| C5{Task too\nbig?}
    C5 -->|Yes| Fix5["Break into steps\nor use Plan mode"]
    C5 -->|No| Fix6["Rewrite prompt\nbefore switching models"]

    style Bad fill:#f8d7da,stroke:#333

Rule of thumb: When an agent fails, assume the instructions failed first — not the model. Rewrite the prompt before switching models.

P. Memory Across Sessions — What Persists vs What Doesn't

Part 1 explained context within a single conversation. But what does the agent remember next time you open a new chat?

What persists (across sessions)

PersistsWhere it livesExample
RulesProject or user settings"Always commit with a descriptive message"
InstructionsAgent / user configuration"Respond in plain English"
SkillsSkills folder"How to write a competitive pulse brief" playbook
Project filesYour repo / DriveCode, docs, configs the agent can read anytime

What does NOT persist

Does not persistWhy it matters
Previous chat historyEach new conversation starts blank
Tool call results from last sessionAgent won't remember "I already searched Jira yesterday"
Decisions you made in an old threadYou must restate preferences unless they're in a Rule

How to give an agent "long-term memory"

Since LLMs don't remember past chats, you simulate memory by writing things down:

  1. Rules — for behavior that should always apply
  2. Skills — for workflows you repeat
  3. Project docs — README.md, AGENTS.md, and other .md notes the agent can read (Markdown keeps them editable in VS Code/Cursor and diffable in Git)
  4. @-mentions — pull specific files into context on demand

Example: Instead of hoping the agent remembers your coding style from last week, add a Rule: "Use 2-space indentation and prefer functional components in React."

Analogy: The agent wakes up with amnesia every session. Rules and skills are the sticky notes on its desk.

Q. Security & Privacy Basics

When an agent reads your files, runs commands, or calls APIs, data leaves your machine and goes to an AI provider (OpenAI, Anthropic, Google, etc.). Understanding what flows where keeps you safe.

What typically gets sent to the AI provider

Sent to the modelUsually stays local
Text in your prompt and conversationFiles the agent never reads
Contents of files the agent opensData on systems without MCP access
Output of tool calls (search results, API responses)Your API keys (stored in local config, not in prompts)

Safe practices

DoDon't
Use .env files for secrets; add them to .gitignorePaste API keys or passwords into chat
Review agent actions before irreversible stepsLet agents auto-post to public channels unchecked
Use Rules to block dangerous commandsGive agents access to production databases without guardrails
Check your company's AI usage policyPut customer PII or regulated data in prompts without approval

MCP and permissions

Each MCP server only exposes the actions you've connected and authorized. The agent can't read Microsoft Teams unless you've set up the Microsoft Teams MCP server and logged in.

Rule of thumb: Treat the agent like a contractor with laptop access. Give it what it needs for the job — not the keys to everything.

R. Evaluating an Agent — Is It Actually Working?

Once you've built a skill or set up a workflow, you need to know if it's reliable — especially before you trust it to run weekly or unattended.

What "good" looks like

SignalGoodBad
ConsistencySame input → same quality output every timeResults vary wildly run to run
CompletenessAll steps in the skill are executedSkips steps silently
AccuracyFacts match source dataInvents ticket numbers, file paths, or quotes
SafetyPauses before irreversible actionsPosts, deletes, or deploys without asking

A simple evaluation method

  1. Pick 3 real examples — past tasks you'd actually give the agent
  2. Run the agent on each — without helping mid-way
  3. Score each run — complete? accurate? safe?
  4. Fix the skill or rules — not just the prompt — based on what failed
  5. Re-run until 3/3 pass

Example: Your "workshop notes → Jira tickets" skill fails because it assigns the wrong person. Fix: add a step in the skill — "Look up assignee email in the team roster before creating the ticket."

Analogy: Evaluating an agent is like a job interview with a work sample. One good answer isn't enough — you need repeatable performance.

Evaluation loop

flowchart LR
    E1[Pick 3 real\nexamples] --> E2[Run agent\nwithout helping]
    E2 --> E3[Score:\ncomplete? accurate? safe?]
    E3 --> Pass{3/3 pass?}
    Pass -->|No| E4[Fix skill\nor rules]
    E4 --> E2
    Pass -->|Yes| Ship[Ready to trust\nor automate]

    style Ship fill:#d4edda,stroke:#333

S. From One Agent to a Workflow — Chaining Steps Together

Real work rarely fits in a single agent call. A workflow is a sequence of tasks — sometimes handled by one agent with a detailed skill, sometimes by multiple specialized agents.

Single agent vs multi-step workflow

ApproachWhen to useExample
One agent, one skillSteps are linear and relatedWorkshop notes → Jira tickets
One agent, multiple skillsTask spans domains but one "owner"Research competitor → write brief → draft executive summary
Multiple agentsSteps are independent or parallelAgent A pulls market data; Agent B writes the intel report

Example workflow: weekly intelligence & metrics digest

Combines Competitive Intelligence Analyst + Data Analyst work.

flowchart LR
    S1["① #competitive-intel channel\n(Microsoft Teams MCP)"] --> S2["② Warehouse / BI export\n(SQL / Looker)"]
    S2 --> S3["③ Cluster themes\n+ write summary"]
    S3 --> S4["④ Save to Drive\n(Drive MCP)"]
    S4 --> S5["⑤ You review\n→ share with leadership"]

    style S5 fill:#fff3cd,stroke:#333

Each step can be one skill. The overall workflow is documented in a master skill or a checklist you run weekly.

Practical rule: If a workflow has more than 5 steps or 3 different tools, write it down as a skill — don't rely on remembering the sequence each time.
